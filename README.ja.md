# AhoBench
AhoBenchは、非常に単純な、しかし実用的なベンチマークプログラムです。

依存関係は標準Cライブラリ以外無く、アーキテクチャ依存のコードもありません。    
高性能なワークステーションから小さなシングルボードコンピューターまで、幅広いデバイスを同一のアルゴリズムでベンチマークします。(ただし、GNU/Linuxを始めとしたUnixライクOSが実行できる場合に限ります。)

このベンチマークプログラムは、作者がDhrystoneベンチマークを様々なデバイスに移植しようとして上手く行かなかった経験と、sysbenchを移植しようと考えたがsysbenchは組み込みデバイスには複雑すぎると考えた事から開発されました。

## コンパイル方法
コンパイルには、C99以降をサポートするコンパイラが必要です。残念ながら、C99をサポートしない古いコンパイラでは使用出来ません。
以下のコマンド1行で完了します。
```
gcc -O0 ahobench.c -o ahobench
```

## 使用方法
```
Usage:
ahobench [-j <THREADS>] [-d <DURATION>]
Options:
-d           : Specify the number of duration in seconds. (Default: 10)
-j           : Specify the number of threads to run benchmark. (Default: actual cpu threads)
-v --version : Print version information.
-h --help    : Print this help.
```

## アルゴリズムの基本概念
このベンチマークは**文字列操作は多くの場合、コンピューターにとって困難な作業である**という基本概念に基いています。つまり、文字列操作を沢山する事でベンチマークを行なうのです。

アルゴリズムは以下の通りです。
1. 2つの64バイトのメモリ領域を確保する。
2. 1つ目に対し、`abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ1234567890`という文字列を書き込む。
3. その書き込んだ文字列の27番目の文字(A)をNULLに置き換える。
4. 1つ目の変数に`123456789abcdefghijklmnopqrstuvwxyz`という文字列を連結する。コンピュータの世界では、0は文字列の終わりと見做されますから、つまり連結後は`abcdefghijklmnopqrstuvwxyz123456789abcdefghijklmnopqrstuvwxyz`という文字列になります。
5. これを2つ目の変数にコピーする。
6. 2つ目の変数から、1つ目の変数に再度コピーする。
7. 両方のメモリ領域を開放する。

非常に簡単なアルゴリズムです。このような簡単な作業は頭の悪い人でも出来ますから、日本語で頭の悪い人を意味する「アホ」ベンチマークと名付けられました。

## BakaBenchとの違い
作者は似たような意味の名前を持つ「BakaBench」というベンチマークを作成しています。

BakaBenchはただひたすらに1を足し続ける事で性能を測ります。これでは、ベンチマークを行なうスレッド数を増やすと同じCPUでもどんどんスコアが上がってしまいます。さらに、コア数が少ない最新のCPUとコア数の多い非常に古いCPUを比較しても、コア数が多い古いCPUの方が高いスコアを出してしまいます。  
これはアルゴリズムが簡単すぎるために、特にパイプラインを搭載したCPUでは非常に多くの回数をこなせてしまう事や、古いCPUと新しいCPUで処理の仕方が変わっておらず単純に多いコア数で処理した方が回数をこなせてしまうために発生します。

一方このAhoBenchでは、先述のようにやや複雑(でも人間にとってはシンプルな)アルゴリズムを処理します。これにより、BakaBenchと異なり実用的なベンチマークが行なえ、おもちゃとしてではなく使えるベンチマークソフトウェアとして使用出来るようになりました。

## Q&A
### アルゴリズムは本当に実用的ですか?
私は少なくともそう考えています。その理由は2つあります。

まず一つは、実行するスレッド数を増やしても、スコアは目に見えて上昇しない事です。これは実際のCPUのスレッド数と実行するスレッド数が同じ時、CPUにとっての負荷が最大になっている事を示します。

もう一つは、同じスレッド数で実行した時、古いCPUのスコアと新しいCPUのスコアを比べると明らかに新しいCPUの方がスコアが高くなっている事です。このことより、このアルゴリズムがそのCPUが出し得る最大の性能を引き出している事を示します。

これらの理由から、このベンチマークは十分に実用的であると考えられます。

### 無限ループに入って終了しなくなった!
これはコンパイラ最適化を有効にしてコンパイルした場合に遭遇します。  

ベンチマークのアルゴリズムはコンパイラからすると無駄な処理であるため、コンパイラはそれを削ってプログラムとしての処理を高速化しようとします。  
この時、おそらくアルゴリズムどころか終了判定のためのコードなど、重要な部分さえ削除されてしまうのだと考えられます。

コンパイルする時は必ず`-O0`オプションを付けて、コンパイラによる最適化を無効にしてください。

### Windowsで使用できますか?
はいでもあり、いいえでもあります。

このプログラムはUnixライクOSを対象としており、Unixにしかない標準Cライブラリに依存しています。このためWindowsのMSVCやMinGW-w64ではビルド出来ません。

Windowsでビルドするには、Cygwinが必要です。まずCygwinをインストールしてから、Cygwinのターミナル内で通常のコンパイル方法と同様にコンパイルしてください。動作するはずです。

しかしながら、Windowsは非常に無駄の多いOSです。常にバックグラウンドでいくつものサービスが動作し続け、さらにNTカーネルはプログラムの処理効率が悪いです。それだけでなく、この実行方法ではCygwinの互換レイヤーが挟まるため、さらに処理性能が低下します。  
Windows上でこのベンチマークを実行しても、それは真のCPUの性能とはなりません。Windowsで実行したスコアは他者に共有しないでください。混乱を招きます。

## 「アホ」とは何か
「アホ」は日本語で頭の悪い人や物事を意味します。  
私のもう一つのベンチマークプログラムである「BakaBench」の「Baka」とはほとんど意味が同じです。  

AhoBenchはBakaBenchに比べるとやや頭の良いベンチマークですが、日本語では実際の所、「アホ」と「バカ」に違いはありません。  
しかし、人間には容易い作業である文字列操作が、コンピューターにとって苦手な点は、人間から見るとコンピューターは「アホ」に見えるでしょう。なので「AhoBench」と名付けられました。

## scoredb.csv
このファイルは、様々なCPUのAhoBenchのスコアを記載したデータベースです。

各プロセッサ毎に、単一コアの最低スコア、単一コアの最高スコア、単一コアあたりの平均スコア、全コア合計のスコアが記載されています。  
ARMのプロセッサのような、慣例的にそれがCPU名ではなくSoC名で呼ばれるような場合はSoC名と、内包するコア名の両方を記載しています。  
また、そのスコアに特記事項がある場合は「Note」列に特記事項が記載されています。例えば、Android上で測定したスコアは、Androidのバッテリー消費制限のために実際のプロセッサの処理能力に比べて低く数値が出る場合があり、その旨が記載されています。

もしあなたがAhoBenchでリストに記載されていないプロセッサのベンチマークを行なったら、是非気軽にスコアとプロセッサの情報をこのファイルに追記してマージリクエストを送ってください! マージリクエストはGitHub、GitLabのどちらでも受け付けています。  
このファイルに追記するためにベンチマークを行なう場合、以下の条件に従ってください。  
- ベンチマークは最低でも3回行い、各値が著しく違わない事を確認してください。実行する度に値が著しく変化する場合、裏でバックグラウンドプロセスが動いているなど、何かベンチマークを邪魔するものがあるかもしれません。それを解決してから値を測定してください。
- 可能な限り、軽量な環境で測定してください。どうしてもそのOSしか動かない場合を除いて、WindowsやmacOSなどの重いOSは避けてください。また、GNU/LinuxではOSが完全に起動する前のinitrd環境内など、バックグラウンドプロセスが殆どない環境が望ましいでしょう。やむを得ず重いOSやプログラムの処理に制限を課すAndroidのような特殊なOSを使用した場合は、特記事項にその旨を記載してください。
- CSVファイルの内容は、「Architecture」列、「Vendor」列、「CPU/SoC」列の順番の優先度で、昇順にソートしてください。また、辞書順ソートではなく自然なソートを行なってください。(LibreOffice Calcにはソートダイアログにそれらの設定項目があります。LibreOffice Calcの使用を強く推奨します。)
- アーキテクチャーの名称は、[https://www.debian.org/ports/](https://www.debian.org/ports/)に記載されているアーキテクチャ名を使用してください(例えばmipselやppc64elなど)。 ただし、amd64やarm64については、名前が似ていて混同しやすいため、代わりにそれぞれ「x86_64」と「aarch64」を使用してください。AhoBenchは「アホ」による「アホ」のための「アホ」なベンチマークですから、「アホ」にとっても読みやすくあるべきです。
